# Product Requirements Document - suoha

**Author:** Snchen5
**Date:** 2026-03-05

## 1. 执行摘要

### 产品愿景

suoha（梭哈）是一个面向线下德州扑克玩家的免费移动端 Web 应用，解决没有实体筹码时的记账结算痛点。核心理念：轻松、放心、简单好用——打开手机浏览器即用，用完即走，不存历史数据。

目标用户为休闲德州扑克玩家（朋友聚会场景，2-8人），通过房间机制实现多人实时筹码同步，让玩家专注于游戏本身。

### 产品差异化

- **完全免费 + 极简**：市面工具多为收费且功能臃肿，suoha 只做记账和结算，不做多余的事
- **即用即走**：Web 应用无需下载，不存储历史数据，每局独立
- **信任优先**：基于朋友间互信的全透明操作模式，总账校验（盈亏之和=0）保证公平
- **允许负数筹码**：去掉"买入"概念，大幅简化操作流程

### 项目分类

- **类型**：移动优先 Web 应用（Vite + React H5 + Java 后端 + WebSocket）
- **领域**：通用工具类
- **复杂度**：低（业务逻辑简单，实时同步为主要技术点）
- **项目上下文**：绿地项目（全新开发）

## 2. 成功标准

### 用户成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 房间创建时间 | ≤ 30 秒 | 从打开页面到生成二维码 |
| 加入房间时间 | ≤ 10 秒 | 扫码/输入房间号到进入房间 |
| 首次使用成功率 | 100% | 新用户完成完整流程（创建/加入→操作→结算） |
| 操作上手时间 | ≤ 5 分钟 | 新手学会基本操作（下注、收回、平分） |

### 业务成功指标

| 指标 | 目标值 | 时间范围 |
|------|--------|----------|
| 核心功能稳定 | 无严重 bug | 3 个月内 |
| 真实牌局验证 | ≥ 3 次 | 3 个月内 |
| 持续稳定运行 | 低维护成本 | 12 个月 |

### 技术成功指标

详见第 7 节非功能需求中的性能（NFR-1）和可靠性（NFR-3）指标。

### MVP 成功标准

- 支持 2-8 人完成一局完整的德州扑克记账和结算
- 创建房间 ≤ 30 秒，实时同步延迟 ≤ 1 秒
- 首次使用成功率 = 100%，总账校验通过率 = 100%
- 掉线后能自动重连并恢复状态

## 3. 用户旅程

### 角色 1：小李（普通玩家）

**背景：** 25-30岁，偶尔参加朋友聚会，玩过几次德州扑克但不太熟练，希望工具简单好用。

**旅程：周末朋友局**

周六晚上，小李和几个朋友在清吧聚会。有人提议玩德州扑克，但没人带筹码。朋友老王说"用 suoha 吧"，把二维码亮出来。小李掏出手机扫码，输入昵称"小李"，3秒进入房间。

他不太记得规则，点了右上角的"规则提示"快速看了一眼牌型大小。游戏开始，他点"50"下注，屏幕上实时显示池底变化。赢了一把后，点"收回池底"，筹码数字立刻更新。

两小时后老王点了结算，小李看到自己赢了120，总账校验显示盈亏之和为0，大家都没异议。关掉浏览器，继续喝酒聊天。

**关键交互：** 扫码加入 → 查看规则（可选）→ 下注/收回/平分 → 查看结算
**情感弧线：** 好奇 → 安心（简单好用）→ 专注游戏 → 满意（结算清晰）

---

### 角色 2：老王（房主）

**背景：** 固定组局者，负责创建房间和管理牌局，希望快速开局、轻松管理。

**旅程：组织一场牌局**

老王打开手机浏览器访问 suoha，登录后点"创建房间"，系统生成6位数字房间号和二维码。他把二维码发到群里，看着玩家一个个加入，房间页面实时显示"3/8人"。

游戏过程中，老王和其他人一样自助操作筹码，不需要额外管理。有人问"池底多少"，大家看一眼屏幕就清楚了。

打了几局后大家准备散场，老王点"结束牌局"，结算页面弹出：每人盈亏一目了然，总账校验通过。大家确认无误，关掉页面，房间自动销毁。

**关键交互：** 创建房间 → 分享二维码 → 正常游戏 → 触发结算
**情感弧线：** 轻松（快速开局）→ 省心（不用管账）→ 放心（结算准确）

---

### 角色 3：小李（边缘场景 - 掉线恢复）

**旅程：游戏中掉线**

游戏进行到一半，小李的手机信号不好，WebSocket 断开了。页面显示"连接中..."，几秒后自动重连成功，筹码和池底状态完整恢复，游戏继续。小李甚至没注意到中断。

**关键交互：** 掉线 → 自动重连 → 状态恢复
**情感弧线：** 短暂焦虑 → 安心（自动恢复）

## 4. 技术需求

### 应用架构

- **类型：** SPA（单页应用）
- **前端：** Vite + React（H5 移动优先）
- **后端：** Java
- **实时通信：** WebSocket
- **临时存储：** Redis（房间状态、掉线恢复）

### 浏览器支持

| 浏览器 | 优先级 | 说明 |
|--------|--------|------|
| 微信内置浏览器 | 高 | 主要分享渠道 |
| Safari (iOS) | 高 | iPhone 用户 |
| Chrome (Android) | 高 | Android 用户 |
| 其他移动浏览器 | 低 | 尽量兼容 |

### 响应式设计

- 移动优先设计，针对 375px-428px 宽度优化
- 不需要桌面端适配

### 性能目标

| 指标 | 目标值 |
|------|--------|
| 首屏加载 | ≤ 2 秒（4G 网络） |
| WebSocket 消息延迟 | ≤ 1 秒 |
| 页面交互响应 | ≤ 100ms |

### 明确不做

- SEO 优化（工具类应用，不需要搜索引擎收录）
- 无障碍访问（WCAG）
- 原生应用功能（推送通知、设备权限等）
- CLI 命令行工具

## 5. 范围与阶段规划

### MVP 策略

**问题解决型 MVP** — 最小可用产品聚焦于解决核心痛点：没有实体筹码时的记账结算。

### MVP 功能范围

| 模块 | 功能 | 优先级 |
|------|------|--------|
| 用户系统 | 手机号+密码登录（不存在则自动注册），同时只能在一个房间 | Must-Have |
| 房间系统 | 创建房间、生成房间号+二维码、扫码/输入房间号加入、最多8人、房主转让、结束牌局、自动销毁 | Must-Have |
| 筹码操作 | 初始0可为负、快捷下注(20/50/100/1000)+自定义、收回池底、平分池底 | Must-Have |
| 池底系统 | 公共池底维护、实时显示池底总额+每人下注明细 | Must-Have |
| 实时同步 | WebSocket推送、掉线重连恢复状态 | Must-Have |
| 结算 | 房主触发、每人盈亏清单、总账校验（盈亏之和=0）、结算后不保留数据 | Must-Have |
| 规则提示 | 德州扑克基本规则查看 | Nice-to-Have |

### 明确不做（Out of Scope）

- 历史记录和数据持久化
- 复杂权限管理
- 牌局逻辑（发牌、比大小等）
- 社交功能（聊天、好友列表）
- 微信登录、短信验证码
- 多房间同时参与
- 其他桌游支持
- 商业化功能

### 未来方向（Phase 2+）

- 基于实际使用反馈的功能优化
- 性能优化
- 可能扩展为通用多人实时记账工具

### 关键风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| WebSocket 连接不稳定 | 游戏中断 | 自动重连 + Redis 状态恢复 |
| 微信浏览器兼容性 | 主要渠道不可用 | 早期测试、降级方案 |
| 并发操作冲突 | 筹码数据不一致 | 服务端统一处理、乐观锁 |

## 7. 非功能需求

### NFR-1：性能

| ID | 需求 | 目标值 |
|----|------|--------|
| NFR-1.1 | 首屏加载时间（4G 网络） | ≤ 2 秒 |
| NFR-1.2 | WebSocket 消息传递延迟 | ≤ 1 秒 |
| NFR-1.3 | 页面交互响应时间 | ≤ 100ms |
| NFR-1.4 | 前端打包体积 | ≤ 500KB（gzip） |

### NFR-2：安全

| ID | 需求 | 说明 |
|----|------|------|
| NFR-2.1 | 密码存储 | 使用 bcrypt 或类似算法加密存储，禁止明文 |
| NFR-2.2 | 通信安全 | HTTPS + WSS 加密传输 |
| NFR-2.3 | 认证令牌 | JWT 或类似机制，设置合理过期时间 |
| NFR-2.4 | 房间隔离 | 玩家只能访问自己所在房间的数据 |

### NFR-3：可靠性

| ID | 需求 | 目标值 |
|----|------|--------|
| NFR-3.1 | WebSocket 连接稳定性 | ≥ 99%（游戏过程中） |
| NFR-3.2 | 掉线自动重连 | ≤ 5 秒内完成重连 |
| NFR-3.3 | 状态恢复完整性 | 重连后 100% 恢复之前状态 |
| NFR-3.4 | 并发操作一致性 | 服务端保证筹码计算的原子性和一致性 |
