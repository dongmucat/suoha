# suoha - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for suoha, decomposing the requirements from the PRD, UX Design, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

**FR-1：用户认证**
- FR-1.1: 用户可以通过手机号+密码进行登录
- FR-1.2: 系统在手机号不存在时自动完成注册
- FR-1.3: 系统限制一个用户同时只能在一个房间中

**FR-2：房间管理**
- FR-2.1: 房主可以创建一个新房间
- FR-2.2: 系统为房间生成唯一的纯数字房间号
- FR-2.3: 系统为房间生成可分享的二维码
- FR-2.4: 玩家可以通过扫描二维码加入房间
- FR-2.5: 玩家可以通过输入房间号加入房间
- FR-2.6: 系统限制房间最多容纳 8 名玩家
- FR-2.7: 系统在房间满员时拒绝新玩家并提示
- FR-2.8: 房主可以将房主权限转让给其他玩家
- FR-2.9: 房主可以触发"结束牌局"进入结算
- FR-2.10: 系统在所有玩家离开后自动销毁房间

**FR-3：筹码操作**
- FR-3.1: 所有玩家进入房间时初始筹码为 0
- FR-3.2: 系统允许玩家筹码为负数
- FR-3.3: 玩家可以通过快捷按钮（20/50/100/1000）下注
- FR-3.4: 玩家可以输入自定义金额下注
- FR-3.5: 下注操作需要玩家确认后生效
- FR-3.6: 玩家可以一键收回全部池底到自己的筹码
- FR-3.7: 玩家可以发起平分池底，选择参与平分的玩家后按人数均分

**FR-4：池底系统**
- FR-4.1: 系统维护一个公共池底，记录所有下注金额
- FR-4.2: 系统实时显示池底总额
- FR-4.3: 系统实时显示每位玩家的下注明细

**FR-5：实时同步**
- FR-5.1: 系统通过 WebSocket 实时推送筹码变动给所有房间内玩家
- FR-5.2: 系统通过 WebSocket 实时推送池底变化给所有房间内玩家
- FR-5.3: 系统在玩家掉线后支持自动重连
- FR-5.4: 系统在重连后恢复玩家之前的完整状态

**FR-6：结算**
- FR-6.1: 房主可以触发结算
- FR-6.2: 系统显示每位玩家的盈亏（正数赢/负数输）
- FR-6.3: 系统执行总账校验（所有玩家盈亏之和 = 0）
- FR-6.4: 结算页面关闭后系统不保留任何数据

**FR-7：辅助功能**
- FR-7.1: 玩家可以查看德州扑克基本规则提示

### NonFunctional Requirements

**NFR-1：性能**
- NFR-1.1: 首屏加载时间（4G 网络）≤ 2 秒
- NFR-1.2: WebSocket 消息传递延迟 ≤ 1 秒
- NFR-1.3: 页面交互响应时间 ≤ 100ms
- NFR-1.4: 前端打包体积 ≤ 500KB（gzip）

**NFR-2：安全**
- NFR-2.1: 密码存储使用 bcrypt 或类似算法加密存储，禁止明文
- NFR-2.2: 通信安全使用 HTTPS + WSS 加密传输
- NFR-2.3: 认证令牌使用 JWT 或类似机制，设置合理过期时间
- NFR-2.4: 房间隔离，玩家只能访问自己所在房间的数据

**NFR-3：可靠性**
- NFR-3.1: WebSocket 连接稳定性 ≥ 99%（游戏过程中）
- NFR-3.2: 掉线自动重连 ≤ 5 秒内完成重连
- NFR-3.3: 状态恢复完整性，重连后 100% 恢复之前状态
- NFR-3.4: 并发操作一致性，服务端保证筹码计算的原子性和一致性

### Additional Requirements

**架构技术要求：**
- 启动模板：使用 Vite 官方 React + TypeScript 模板初始化前端项目
- 后端框架：Spring Boot 3.x + Redis + netty-socketio
- 设计系统：Tailwind CSS + shadcn/ui 配置和初始化
- 移动端适配：流式字体（clamp）+ vw 单位 + 固定触摸目标（≥44px）
- 数据结构：Redis Hash + Sorted Set 混合结构
- 并发控制：Lua 脚本保证原子操作

**UX 技术要求：**
- 浏览器支持：微信内置浏览器、Safari (iOS)、Chrome (Android)
- 响应式设计：375px-428px 宽度优化（移动优先）
- 触摸友好：最小触摸目标 44px × 44px
- 实时反馈：数字动画、按钮效果、操作提示
- 颜色系统：主色 #2563EB（蓝色）、成功色 #16A34A（绿色）、警告色 #EF4444（红色）
- 字体系统：系统字体 + 等宽数字字体

## Epic 1: 项目基础设施与用户认证

用户可以注册/登录系统，开发团队拥有完整的前后端项目结构和开发环境。

### Story 1.1: 前端项目初始化

As a 开发者,
I want 使用 Vite + React + TypeScript 模板初始化前端项目，并配置 Tailwind CSS 和 shadcn/ui,
So that 团队拥有标准化的前端开发环境和设计系统基础。

**Acceptance Criteria:**

**Given** 开发者克隆了项目仓库
**When** 运行前端项目启动命令
**Then** Vite 开发服务器成功启动，显示默认页面
**And** Tailwind CSS 样式正常生效
**And** shadcn/ui 组件库可正常引入使用

**Given** 前端项目已初始化
**When** 检查项目配置
**Then** TypeScript 严格模式已启用
**And** 移动端适配基础配置已就绪（375px-428px 视口优化）
**And** 颜色系统变量已定义（主色 #2563EB、成功色 #16A34A、警告色 #EF4444）

### Story 1.2: 后端项目初始化

As a 开发者,
I want 使用 Spring Boot 3.x 初始化后端项目，并配置 Redis 和 netty-socketio,
So that 团队拥有标准化的后端开发环境，支持实时通信和缓存。

**Acceptance Criteria:**

**Given** 开发者克隆了项目仓库
**When** 运行后端项目启动命令
**Then** Spring Boot 应用成功启动
**And** Redis 连接正常
**And** netty-socketio 服务在配置端口（9092）监听

**Given** 后端项目已初始化
**When** 检查项目结构
**Then** 分层架构目录已创建（controller/service/repository/config/model/socket）
**And** 环境配置文件已就绪（application.yml / application-dev.yml）
**And** 全局异常处理和统一响应格式已配置

### Story 1.3: 用户注册与登录

As a 玩家,
I want 通过手机号和密码登录，如果手机号不存在则自动注册,
So that 我可以快速进入系统开始使用，无需繁琐的注册流程。

**Acceptance Criteria:**

**Given** 玩家输入了一个未注册的手机号和密码
**When** 玩家点击登录按钮
**Then** 系统自动创建新账户
**And** 密码使用 bcrypt 加密存储
**And** 返回 JWT 令牌
**And** 玩家自动登录进入首页

**Given** 玩家输入了已注册的手机号和正确密码
**When** 玩家点击登录按钮
**Then** 系统验证密码正确
**And** 返回 JWT 令牌
**And** 玩家登录进入首页

**Given** 玩家输入了已注册的手机号和错误密码
**When** 玩家点击登录按钮
**Then** 系统提示"手机号或密码错误"
**And** 不泄露该手机号是否已注册

**Given** 玩家的 JWT 令牌已过期
**When** 玩家发起任何请求
**Then** 系统返回 401 未授权
**And** 前端引导玩家重新登录

### Story 1.4: 登录页面 UI

As a 玩家,
I want 看到一个简洁美观的移动端登录页面,
So that 我可以快速理解如何登录并开始使用。

**Acceptance Criteria:**

**Given** 玩家打开应用
**When** 页面加载完成
**Then** 显示手机号输入框、密码输入框和登录按钮
**And** 页面适配 375px-428px 移动端屏幕
**And** 触摸目标尺寸 ≥ 44px × 44px
**And** 输入框支持手机键盘数字键盘（手机号）和密码键盘

**Given** 玩家未填写必填字段
**When** 玩家点击登录按钮
**Then** 显示对应字段的验证提示
**And** 手机号格式校验（11位数字）

### Story 1.5: 单房间限制

As a 系统,
I want 限制每个用户同时只能在一个房间中,
So that 避免用户同时参与多个牌局导致数据混乱。

**Acceptance Criteria:**

**Given** 玩家已经在一个房间中
**When** 玩家尝试加入另一个房间
**Then** 系统拒绝加入请求
**And** 提示"您已在房间 XXXXXX 中，请先退出当前房间"

**Given** 玩家已经在一个房间中
**When** 玩家尝试创建新房间
**Then** 系统拒绝创建请求
**And** 提示"您已在房间 XXXXXX 中，请先退出当前房间"

## Epic 2: 房间创建与加入

房主可以创建房间并分享，玩家可以扫码或输入房间号加入房间，所有人看到房间内的玩家列表。

### Story 2.1: 创建房间

As a 玩家,
I want 创建一个新的游戏房间并获得房间号和二维码,
So that 我可以邀请朋友加入一起玩牌。

**Acceptance Criteria:**

**Given** 玩家已登录且不在任何房间中
**When** 玩家点击"创建房间"按钮
**Then** 系统创建新房间并生成唯一的 6 位纯数字房间号
**And** 系统生成该房间的二维码（包含加入链接）
**And** 玩家自动成为房主并进入房间页面
**And** 玩家初始筹码为 0
**And** WebSocket 连接建立成功

**Given** 房间已创建
**When** 房主查看房间信息
**Then** 显示房间号和二维码
**And** 二维码可供其他玩家扫描加入

### Story 2.2: 输入房间号加入房间

As a 玩家,
I want 通过输入 6 位房间号加入已有房间,
So that 我可以在朋友告知房间号后快速加入牌局。

**Acceptance Criteria:**

**Given** 玩家已登录且不在任何房间中
**When** 玩家输入有效的 6 位房间号并点击"加入"
**Then** 玩家成功加入房间
**And** 玩家初始筹码为 0
**And** WebSocket 连接建立成功
**And** 房间内所有玩家实时看到新玩家加入

**Given** 玩家输入了不存在的房间号
**When** 玩家点击"加入"
**Then** 系统提示"房间不存在，请检查房间号"

**Given** 目标房间已满 8 人
**When** 玩家尝试加入
**Then** 系统拒绝加入并提示"房间已满，无法加入"

### Story 2.3: 扫码加入房间

As a 玩家,
I want 通过扫描二维码直接加入房间,
So that 我可以在面对面场景下最快速地加入牌局。

**Acceptance Criteria:**

**Given** 玩家已登录且不在任何房间中
**When** 玩家扫描房间二维码
**Then** 系统解析二维码中的房间信息
**And** 玩家自动加入对应房间
**And** 玩家初始筹码为 0
**And** WebSocket 连接建立成功

**Given** 玩家未登录
**When** 玩家扫描房间二维码
**Then** 系统引导玩家先登录
**And** 登录成功后自动加入对应房间

**Given** 二维码对应的房间已满或不存在
**When** 玩家扫描二维码
**Then** 系统显示对应的错误提示

### Story 2.4: 房间页面与玩家列表

As a 玩家,
I want 在房间页面看到所有玩家的列表和基本信息,
So that 我知道谁在房间里以及每个人的状态。

**Acceptance Criteria:**

**Given** 玩家已进入房间
**When** 房间页面加载完成
**Then** 显示房间号（顶部栏）
**And** 显示所有在线玩家的列表（昵称、筹码数）
**And** 房主标识清晰可见
**And** 连接状态指示器显示"● 已连接"（绿色）

**Given** 有新玩家加入房间
**When** 新玩家加入成功
**Then** 玩家列表实时更新，显示新玩家
**And** 显示 toast 提示"XXX 加入了房间"

**Given** 有玩家离开房间
**When** 玩家退出
**Then** 玩家列表实时更新，移除该玩家
**And** 显示 toast 提示"XXX 离开了房间"

### Story 2.5: 首页 UI（创建/加入入口）

As a 玩家,
I want 在首页看到清晰的"创建房间"和"加入房间"入口,
So that 我可以快速选择创建或加入牌局。

**Acceptance Criteria:**

**Given** 玩家已登录
**When** 进入首页
**Then** 显示"创建房间"按钮和"加入房间"入口（房间号输入框 + 加入按钮）
**And** 页面布局适配移动端（375px-428px）
**And** 按钮触摸目标 ≥ 44px × 44px
**And** 如果玩家已在某个房间中，显示"回到房间"入口

## Epic 3: 筹码操作与池底管理

玩家可以下注到池底、收回池底、平分池底，所有操作实时同步给房间内所有人。

### Story 3.1: 快捷下注

As a 玩家,
I want 通过快捷按钮（20/50/100/1000）快速下注到池底,
So that 我可以在牌局中高效地完成最常见的下注操作。

**Acceptance Criteria:**

**Given** 玩家在房间中
**When** 玩家点击快捷下注按钮（20/50/100/1000）
**Then** 弹出确认提示
**And** 玩家确认后，下注金额从玩家筹码扣除并加入池底
**And** 玩家筹码可以变为负数
**And** 池底总额实时更新
**And** 房间内所有玩家通过 WebSocket 实时看到筹码和池底变化

**Given** 玩家点击快捷下注按钮后弹出确认
**When** 玩家取消确认
**Then** 不执行任何操作，筹码和池底不变

### Story 3.2: 自定义金额下注

As a 玩家,
I want 输入自定义金额进行下注,
So that 我可以在快捷按钮不满足需求时灵活下注。

**Acceptance Criteria:**

**Given** 玩家在房间中
**When** 玩家输入自定义金额并点击下注
**Then** 弹出确认提示
**And** 玩家确认后，下注金额从玩家筹码扣除并加入池底
**And** 房间内所有玩家实时看到变化

**Given** 玩家输入了无效金额（0、负数、非数字）
**When** 玩家点击下注
**Then** 系统提示输入有效金额
**And** 不执行下注操作

### Story 3.3: 收回池底

As a 玩家,
I want 一键收回全部池底到自己的筹码,
So that 赢得一手牌后可以快速收回所有筹码。

**Acceptance Criteria:**

**Given** 池底总额大于 0
**When** 玩家点击"收回池底"按钮
**Then** 池底全部金额加到该玩家的筹码上
**And** 池底归零
**And** 所有下注明细清空
**And** 房间内所有玩家实时看到变化

**Given** 池底总额为 0
**When** 玩家点击"收回池底"按钮
**Then** 系统提示"池底为空，无法收回"
**And** 不执行任何操作

### Story 3.4: 平分池底

As a 玩家,
I want 发起平分池底并选择参与平分的玩家,
So that 多人赢得一手牌时可以公平分配池底。

**Acceptance Criteria:**

**Given** 池底总额大于 0
**When** 玩家点击"平分池底"按钮
**Then** 显示玩家选择界面，列出房间内所有玩家（含自己）
**And** 玩家勾选参与平分的人后确认

**Given** 玩家已选择参与平分的人并确认
**When** 系统执行平分
**Then** 池底金额按参与人数均分，分配到各玩家筹码
**And** 如有余数，余数保留在池底
**And** 池底更新为余数或归零
**And** 房间内所有玩家实时看到变化

**Given** 池底总额为 0
**When** 玩家点击"平分池底"
**Then** 系统提示"池底为空，无法平分"

### Story 3.5: 池底显示与下注明细

As a 玩家,
I want 实时看到池底总额和每位玩家的下注明细,

**Then** 池底区域显示当前池底总额（48px 等宽字体）

**Given** 任意玩家执行了下注操作
**When** WebSocket 推送到达
**Then** 池底总额数字动画过渡更新（0.3s ease-out）
**And** 对应玩家的下注明细实时更新
**And** 对应玩家的筹码数字实时更新

**Given** 池底被收回或平分
**When** 操作完成
**Then** 池底总额归零（或显示余数）
**And** 下注明细清空
**And** 相关玩家筹码更新

### Story 3.6: 筹码操作区 UI

As a 玩家,
I want 在房间底部看到清晰的操作区域，包含快捷按钮和操作按钮,
So that 我可以单手快速完成所有筹码操作。

**Acceptance Criteria:**

**Given** 玩家在房间中

**Then** 快捷下注按钮（20/50/100/1000）水平排列，使用强调色 #F59E0B
**And** 自定义金额输入框和下注按钮可见
**And** "收回池底"和"平分池底"按钮可见

## Epic 4: 掉线恢复与连接稳定性

玩家掉线后自动重连，完整恢复游戏状态，游戏不中断。

### Story 4.1: 掉线检测与自动重连

As a 玩家,
I want 在网络断开后系统自动尝试重连,
So that 我不需要手动刷新页面就能恢复游戏。

**Acceptance Criteria:**

**Given** 玩家在房间中且 WebSocket 连接断开
**When** 系统检测到掉线
**Then** 连接状态指示器变为"● 重连中"（黄色）
**And** 系统自动尝试重连（指数退避策略）
**And** 5 秒内完成重连

**Given** 自动重连成功
**When** WebSocket 连接恢复
**Then** 连接状态指示器恢复为"● 已连接"（绿色）
**And** 显示 toast 提示"已重新连接"

**Given** 多次重连失败（超过 30 秒）
**When** 系统无法恢复连接
**Then** 显示"网络连接失败，请检查网络后点击重试"
**And** 提供手动重试按钮

### Story 4.2: 状态恢复

As a 玩家,
I want 重连后完整恢复之前的游戏状态,
So that 掉线不会影响我的游戏体验和数据完整性。

**Acceptance Criteria:**

**Given** 玩家掉线后重连成功
**When** 系统恢复状态
**Then** 玩家的筹码数据与掉线前一致
**And** 池底总额和下注明细与当前服务端状态一致
**And** 玩家列表与当前房间状态一致
**And** 房主标识正确显示

**Given** 玩家掉线期间其他玩家执行了操作
**When** 掉线玩家重连
**Then** 系统推送最新的完整房间状态快照
**And** 玩家看到的数据与其他在线玩家完全一致

## Epic 5: 结算与房间管理

房主可以结束牌局、查看结算、转让权限，房间在结束后自动销毁。

### Story 5.1: 房主转让

As a 房主,
I want 将房主权限转让给房间内的其他玩家,
So that 我需要离开时牌局可以继续进行。

**Acceptance Criteria:**

**Given** 当前用户是房主且房间内有其他玩家
**When** 房主点击某个玩家并选择"转让房主"
**Then** 弹出确认提示
**And** 确认后该玩家成为新房主
**And** 房间内所有玩家实时看到房主标识变更
**And** 显示 toast 提示"XXX 已成为房主"

**Given** 房间内只有房主一人
**When** 房主尝试转让
**Then** 系统提示"房间内没有其他玩家，无法转让"

### Story 5.2: 结束牌局与结算

As a 房主,
I want 结束牌局并查看所有玩家的盈亏结算,
So that 大家可以清楚地知道每个人赢了多少或输了多少。

**Acceptance Criteria:**

**Given** 当前用户是房主
**When** 房主点击"结束牌局"按钮
**Then** 弹出确认提示"确定结束牌局？结束后将进入结算页面"
**And** 确认后所有玩家进入结算页面

**Given** 进入结算页面
**When** 页面加载完成
**Then** 显示每位玩家的最终盈亏（正数绿色表示赢，负数红色表示输）
**And** 系统执行总账校验（所有玩家盈亏之和 = 0）
**And** 校验通过时显示"✅ 总账校验通过"
**And** 校验不通过时显示"⚠️ 总账异常，请联系管理员"

**Given** 池底不为零时
**When** 房主尝试结束牌局
**Then** 系统提示"池底还有 XXX 未分配，请先收回或平分池底"
**And** 阻止结算

### Story 5.3: 结算后数据清理与房间销毁

As a 系统,
I want 在结算完成后清理所有房间数据,
So that 不保留任何历史数据，保护用户隐私。

**Acceptance Criteria:**

**Given** 所有玩家已查看结算结果
**When** 玩家关闭结算页面或点击"返回首页"
**Then** 玩家退出房间并返回首页
**And** WebSocket 连接断开

**Given** 房间内所有玩家都已离开
**When** 系统检测到房间为空
**Then** Redis 中该房间的所有数据被删除
**And** 房间号释放

**Given** 牌局结束但有玩家未主动离开
**When** 超过设定的超时时间
**Then** 系统自动清理房间数据
**And** 强制断开剩余连接
