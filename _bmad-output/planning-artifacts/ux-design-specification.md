## 核心用户体验

### 定义体验

suoha 的核心体验围绕**实时筹码操作**展开。用户最频繁的操作是：下注到池底、收回池底、查看实时筹码变化。整个产品的价值建立在一个关键交互上：**所有玩家看到完全一致的筹码和池底状态**。这种实时同步是信任的基础，也是产品与传统记账方式的本质区别。

核心用户循环：
1. 快速下注（快捷按钮或自定义金额）
2. 实时看到池底和所有人的筹码变化
3. 一键收回池底或平分
4. 继续下一轮

这个循环必须毫不费力、零延迟、完全可靠。

### 平台策略

**主平台：移动端 Web 应用（H5）**

- **目标浏览器**：微信内置浏览器（主要分享渠道）、Safari (iOS)、Chrome (Android)
- **交互方式**：触摸操作为主，针对手机屏幕优化
- **屏幕尺寸**：375px-428px 宽度（移动优先，不考虑桌面端）
- **设备能力**：
  - 摄像头（扫码加入房间）
  - WebSocket 实时通信
  - 不需要离线功能，但必须处理掉线重连

**平台约束：**
- 无需下载 App，打开浏览器即用
- 不需要 SEO 优化（工具类应用）
- 不考虑无障碍访问（WCAG）

### 毫不费力的交互

以下交互必须做到零思考、零摩擦：

1. **加入房间**：扫码 3 秒进入，或输入 6 位数字房间号
2. **快速下注**：快捷按钮（20/50/100/1000）覆盖 90% 场景，点击即下注
3. **一键收回**：赢了池底，一键收回全部筹码
4. **实时同步**：所有操作自动推送给房间内所有人，无需刷新
5. **自动重连**：掉线后 5 秒内自动重连，状态完整恢复
6. **总账校验**：结算时自动校验盈亏之和=0，无需手动核对

**消除的步骤：**
- 去掉"买入"概念：允许负数筹码，简化操作流程
- 去掉下载 App：Web 应用即用即走
- 去掉复杂权限：基于朋友间互信，全透明操作

### 关键成功时刻

**"这更好"的时刻：**
- 扫码进入房间只需 3 秒，不用下载 App，不用注册（手机号自动注册）
- 实时看到所有人的操作，不用问"池底多少"

**用户感到成功的时刻：**
- 结算时总账校验通过（盈亏之和=0），大家都没异议
- 掉线后自动重连，状态完整恢复，游戏继续

**决定成败的交互：**
- **实时同步失败**：如果看到的筹码不一致，信任崩塌
- **掉线无法恢复**：如果掉线后无法恢复状态，游戏中断
- **结算错误**：如果总账校验不通过，产品失去价值

**首次使用成功：**
新用户完成第一次完整流程（加入房间 → 下注/收回 → 结算），并且：
- 加入房间 ≤ 10 秒
- 实时同步延迟 ≤ 1 秒
- 结算总账校验通过

### 体验原则

以下原则将指导所有 UX 设计决策：

1. **即时透明** — 所有操作实时同步，所有人看到一致的状态，建立信任基础。任何延迟超过 1 秒的操作都需要优化。

2. **零门槛进入** — 扫码 3 秒进入，不下载、不注册（手机号自动注册），用完即走。首次使用成功率必须达到 100%。

3. **快捷操作优先** — 快捷按钮覆盖 90% 场景，自定义输入作为补充。每个常用操作不超过 2 次点击。

4. **自动容错** — 掉线自动重连、状态自动恢复、总账自动校验，用户无需担心技术问题。系统主动处理异常，而不是让用户处理。

## 情感响应设计

### 目标情感状态

suoha 的设计目标是让用户在整个使用过程中感到**放心、高效、专注**。

**核心情感目标：**

1. **信任感** — 用户相信系统显示的数据是准确的、实时的、一致的
   - 所有人看到相同的筹码和池底状态
   - 结算时总账自动校验通过
   - 掉线后状态完整恢复

2. **掌控感** — 用户感觉自己完全掌控自己的筹码操作
   - 快捷按钮响应迅速（≤100ms）
   - 操作可撤销（收回池底）
   - 实时看到操作结果

3. **专注游戏** — 用户可以专注于德州扑克游戏本身，而不是记账工具
   - 操作简单到不需要思考
   - 界面不干扰游戏进程
   - 自动处理技术问题（重连、校验）

**避免的情感：**
- 焦虑：担心数据不准确、担心掉线丢失状态
- 困惑：不知道如何操作、不确定操作是否成功
- 挫败：操作复杂、需要多次尝试

### 情感旅程映射

#### 场景 1：首次使用（新用户小李）

**阶段 1：加入房间**
- **情感曲线**：好奇 → 轻松惊喜
- **触发点**：扫码进入房间只需 3 秒
- **设计响应**：
  - 扫码后立即显示房间信息（房间号、房主、当前人数）
  - 自动注册无需填写表单
  - 清晰的欢迎提示："已加入房间 123456"

**阶段 2：首次下注**
- **情感曲线**：尝试 → 确认成功
- **触发点**：点击快捷按钮下注 100
- **设计响应**：
  - 按钮点击后立即视觉反馈（按下效果）
  - 筹码数字动画变化（-100）
  - 池底数字动画增加（+100）
  - 所有人同时看到变化

**阶段 3：观察实时同步**
- **情感曲线**：观察 → 建立信任
- **触发点**：看到其他人的操作实时显示
- **设计响应**：
  - 其他人下注时，显示操作提示："老王下注 50"
  - 筹码和池底数字实时更新
  - 延迟 ≤ 1 秒

**阶段 4：首次结算**
- **情感曲线**：期待 → 满意
- **触发点**：房主点击结算，总账校验通过
- **设计响应**：
  - 显示每个人的盈亏清单
  - 自动校验盈亏之和=0，显示"总账校验通过 ✓"
  - 清晰的结算结果，无争议

#### 场景 2：掉线恢复（普通玩家）

**阶段 1：掉线**
- **情感曲线**：正常 → 焦虑
- **触发点**：网络中断，连接断开
- **设计响应**：
  - 显示"连接中断，正在重连..."提示
  - 不清空界面数据，保持最后状态可见
  - 自动尝试重连，无需用户操作

**阶段 2：重连中**
- **情感曲线**：焦虑 → 期待
- **触发点**：系统自动重连
- **设计响应**：
  - 显示重连进度："正在重连... (3/5)"
  - 5 秒内完成重连
  - 重连成功后显示"已重新连接"

**阶段 3：状态恢复**
- **情感曲线**：期待 → 放心
- **触发点**：重连成功，状态完整恢复
- **设计响应**：
  - 自动同步最新状态（筹码、池底、房间成员）
  - 显示"状态已恢复，游戏继续"
  - 无需手动刷新或重新加入

#### 场景 3：房主管理房间

**阶段 1：创建房间**
- **情感曲线**：主动 → 顺利
- **触发点**：点击"创建房间"
- **设计响应**：
  - 立即生成房间号和二维码
  - 清晰的分享提示："分享二维码邀请朋友加入"
  - 显示当前房间人数（1/8）

**阶段 2：管理池底**
- **情感曲线**：掌控 → 高效
- **触发点**：需要平分池底或收回池底
- **设计响应**：
  - 房主专属操作按钮："平分池底"、"收回池底"
  - 操作前显示确认提示："确认平分池底 500 给 4 人？"
  - 操作后显示结果："已平分，每人 +125"

**阶段 3：结束牌局**
- **情感曲线**：决策 → 确认
- **触发点**：点击"结束牌局"
- **设计响应**：
  - 显示结算清单（每个人的盈亏）
  - 自动校验总账（盈亏之和=0）
  - 确认后房间自动销毁，数据不保留

### 情感设计原则

1. **即时反馈** — 每个操作都有立即的视觉反馈，让用户确认操作成功
   - 按钮点击效果
   - 数字动画变化
   - 操作提示消息

2. **透明状态** — 用户始终清楚当前状态和系统正在做什么
   - 实时显示连接状态
   - 重连进度提示
   - 操作结果确认

3. **主动容错** — 系统主动处理异常，而不是让用户处理
   - 自动重连
   - 自动状态恢复
   - 自动总账校验

4. **减少焦虑** — 通过设计消除用户的担忧
   - 掉线不丢失状态
   - 结算自动校验
   - 操作可撤销


## UX Pattern Analysis & Inspiration

### Inspiring Products Analysis

基于 suoha 的核心场景（线下德州扑克记账、实时同步、扫码加入），我们分析以下优秀产品的 UX 设计：

**1. 微信 AA 收款**

**核心问题解决**：朋友间快速分摊费用，建立信任和透明
- **做得好的地方**：
  - 扫码快速加入，无需下载或注册
  - 实时看到谁付了钱，所有人看到一致状态
  - 自动计算分摊金额，无需手动核对
  - 操作简单，快捷按钮覆盖常用金额
  
- **情感设计成功之处**：
  - **信任感**：所有人看到相同的付款记录，透明公开
  - **掌控感**：清楚看到自己的付款状态和欠款金额
  - **高效感**：快捷操作，不需要思考

- **创新交互**：
  - 扫码进入 → 自动加入收款群组
  - 实时推送 → 有人付款立即通知
  - 一键支付 → 点击即完成付款

**2. 支付宝群收款**

**核心问题解决**：多人收款场景，快速高效完成支付
- **做得好的地方**：
  - 快捷金额按钮（10/20/50/100），覆盖 90% 场景
  - 实时推送通知，所有人同步看到收款进度
  - 清晰的收款记录，支持查看明细
  - 自动计算总额和未付人数
  
- **情感设计成功之处**：
  - **放心感**：清晰的收款记录，不会遗漏
  - **专注感**：快捷操作让用户专注于社交，而不是工具
  - **成就感**：看到收款进度条完成，满足感

- **创新交互**：
  - 快捷金额按钮 → 点击即支付
  - 实时进度条 → 可视化收款进度
  - 自动提醒 → 未付款用户自动收到提醒

**3. 微信群聊（实时通信）**

**核心问题解决**：多人实时通信，保证消息不丢失
- **做得好的地方**：
  - 消息实时同步，所有人看到一致的聊天记录
  - 掉线自动重连，5 秒内恢复连接
  - 状态恢复，重连后自动同步未读消息
  - 离线消息推送，不错过重要信息
  
- **情感设计成功之处**：
  - **信任感**：消息不会丢失，所有人看到一致状态
  - **放心感**：掉线后自动重连，无需担心
  - **专注感**：自动处理技术问题，用户专注于沟通

- **创新交互**：
  - WebSocket 实时推送 → 消息延迟 ≤ 1 秒
  - 自动重连机制 → 掉线后 5 秒内恢复
  - 状态持久化 → 重连后完整恢复聊天记录

### Transferable UX Patterns

从以上产品中提取可应用到 suoha 的 UX 模式：

**导航与信息架构模式**

1. **扫码快速进入模式**（来自微信 AA 收款）
   - 适用场景：suoha 的加入房间流程
   - 实现方式：扫描二维码 → 自动注册 → 立即进入房间
   - 预期效果：3 秒完成加入，零门槛

2. **单屏操作模式**（来自支付宝群收款）
   - 适用场景：suoha 的筹码操作界面
   - 实现方式：所有核心操作在一个屏幕完成，无需跳转
   - 预期效果：减少认知负担，专注游戏

**交互模式**

1. **快捷按钮优先模式**（来自支付宝群收款）
   - 适用场景：suoha 的下注操作
   - 实现方式：快捷按钮 20/50/100/1000，覆盖 90% 场景
   - 预期效果：点击即下注，≤ 2 次操作

2. **实时同步模式**（来自微信群聊）
   - 适用场景：suoha 的筹码和池底状态同步
   - 实现方式：WebSocket 推送，所有人看到一致状态
   - 预期效果：建立信任，延迟 ≤ 1 秒

3. **自动重连模式**（来自微信群聊）
   - 适用场景：suoha 的掉线恢复
   - 实现方式：掉线后自动尝试重连，5 秒内恢复状态
   - 预期效果：减少焦虑，游戏继续

4. **操作即时反馈模式**（来自支付宝群收款）
   - 适用场景：suoha 的所有筹码操作
   - 实现方式：按钮点击效果 + 数字动画变化 + 操作提示
   - 预期效果：确认操作成功，建立掌控感

**视觉设计模式**

1. **数字突出显示模式**（来自支付宝群收款）
   - 适用场景：suoha 的筹码和池底显示
   - 实现方式：大字号、高对比度、动画变化
   - 预期效果：一眼看清关键数据

2. **状态指示器模式**（来自微信群聊）
   - 适用场景：suoha 的连接状态显示
   - 实现方式：顶部状态栏显示"已连接"/"重连中"
   - 预期效果：透明状态，减少焦虑

3. **操作历史流模式**（来自微信 AA 收款）
   - 适用场景：suoha 的操作记录显示
   - 实现方式：时间线显示所有人的操作记录
   - 预期效果：建立信任，可追溯

### Anti-Patterns to Avoid

从分析中识别出需要避免的 UX 反模式：

**导航反模式**

1. **多步骤注册流程** — 用户觉得繁琐，放弃使用
   - 为什么避免：suoha 的目标是"即用即走"，复杂注册违背核心价值
   - 替代方案：手机号自动注册，扫码 3 秒进入

2. **多页面跳转** — 用户容易迷失，增加认知负担
   - 为什么避免：德州扑克游戏节奏快，频繁跳转会打断游戏
   - 替代方案：单屏操作，所有核心功能在一个界面

**交互反模式**

1. **需要手动刷新** — 用户不确定数据是否最新
   - 为什么避免：实时同步是 suoha 的核心价值，手动刷新会破坏信任
   - 替代方案：WebSocket 自动推送，实时更新

2. **复杂的输入表单** — 用户觉得麻烦，操作慢
   - 为什么避免：德州扑克游戏节奏快，复杂输入会拖慢游戏
   - 替代方案：快捷按钮优先，自定义输入作为补充

3. **没有操作反馈** — 用户不确定操作是否成功
   - 为什么避免：不确定性会导致焦虑和重复操作
   - 替代方案：即时视觉反馈（按钮效果 + 数字动画 + 提示消息）

**视觉反模式**

1. **信息过载** — 用户被大量信息淹没，找不到重点
   - 为什么避免：德州扑克游戏需要专注，过多信息会分散注意力
   - 替代方案：突出关键数据（筹码、池底），隐藏次要信息

2. **不一致的视觉语言** — 用户困惑，学习成本高
   - 为什么避免：工具类应用需要快速上手，不一致会增加学习成本
   - 替代方案：统一的设计系统，一致的交互模式

**技术反模式**

1. **掉线后需要手动重连** — 用户焦虑，游戏中断
   - 为什么避免：线下游戏场景网络不稳定，手动重连会打断游戏
   - 替代方案：自动重连，5 秒内恢复状态

2. **状态不恢复** — 用户担心数据丢失
   - 为什么避免：筹码数据是核心，丢失会导致信任崩塌
   - 替代方案：状态持久化，重连后完整恢复

### Design Inspiration Strategy

基于以上分析，制定 suoha 的设计灵感策略：

**直接采用的模式**

1. **扫码快速进入**（微信 AA 收款）
   - 原因：完美匹配 suoha 的"零门槛进入"原则
   - 实现：扫描二维码 → 自动注册 → 3 秒进入房间

2. **快捷按钮优先**（支付宝群收款）
   - 原因：支持 suoha 的"快捷操作优先"原则
   - 实现：快捷按钮 20/50/100/1000，覆盖 90% 下注场景

3. **实时同步**（微信群聊）
   - 原因：suoha 的核心价值，建立信任的基础
   - 实现：WebSocket 推送，所有人看到一致状态，延迟 ≤ 1 秒

4. **自动重连**（微信群聊）
   - 原因：支持 suoha 的"自动容错"原则
   - 实现：掉线后自动重连，5 秒内恢复状态

**需要调整的模式**

1. **操作历史流**（微信 AA 收款）
   - 调整原因：德州扑克游戏节奏快，完整历史流会占用屏幕空间
   - 调整方案：简化为最近 3 条操作记录，支持展开查看完整历史

2. **自动提醒**（支付宝群收款）
   - 调整原因：suoha 基于朋友间互信，不需要催促式提醒
   - 调整方案：仅在结算时显示盈亏清单，不主动催促

**明确避免的模式**

1. **多步骤注册流程** — 违背"零门槛进入"原则
2. **手动刷新** — 违背"实时同步"核心价值
3. **复杂输入表单** — 违背"快捷操作优先"原则
4. **掉线需要手动重连** — 违背"自动容错"原则

**设计决策指导原则**

当面临设计决策时，优先考虑：
1. 是否支持"即时透明"原则？（实时同步 ≤ 1 秒）
2. 是否支持"零门槛进入"原则？（3 秒加入房间）
3. 是否支持"快捷操作优先"原则？（≤ 2 次点击）
4. 是否支持"自动容错"原则？（自动重连、状态恢复）

这个策略将指导 suoha 的所有设计决策，同时保持产品的独特性。

## 设计系统选择

### 选定方案：Tailwind CSS + shadcn/ui

基于 suoha 的项目需求（移动优先、快速落地、低复杂度、工具类应用），选择 **Tailwind CSS + shadcn/ui** 作为设计系统基础。

**选择理由：**

1. **快速开发** — Tailwind 的 utility-first 方法快速构建 UI，无需写大量 CSS
2. **移动优先** — 天然支持响应式设计，完美匹配移动端 H5 策略
3. **轻量灵活** — shadcn/ui 提供可复制的组件代码，完全可控
4. **易于定制** — 通过 Tailwind 配置轻松定制颜色、间距、字体
5. **性能优秀** — 生产环境自动清除未使用的 CSS，包体积小

### 实现方式

- Tailwind CSS 作为基础样式系统
- shadcn/ui 组件作为起点（Button, Input, Dialog 等）
- 根据 suoha 设计需求定制主题
- 创建自定义组件（筹码显示、池底显示、快捷按钮等）

### 定制策略

**颜色系统**
- 主色调：信任感的蓝色/绿色（传达放心、透明）
- 强调色：操作按钮（下注、收回、平分）
- 成功色：总账校验通过、赢钱
- 警告色：负数筹码、掉线状态
- 中性色：背景、文字、分割线

**字体系统**
- 数字：等宽字体（清晰易读，筹码和池底数字）
- 文字：系统字体（性能优化，减少加载时间）
- 字号层级：大（筹码/池底数字）→ 中（操作按钮）→ 小（辅助信息）

**间距系统**
- 触摸目标尺寸 ≥ 44px（移动端优化）
- 按钮间距足够大，防止误触
- 信息层级通过间距区分

**动画系统**
- 数字变化动画（筹码增减、池底变化）
- 按钮反馈动画（点击效果）
- 状态切换动画（连接/断开、加入/离开）
- 保持动画简洁，不影响性能

## 定义核心体验

### 标志性交互

> **"扫码进入，点击下注，所有人实时看到"**

这是 suoha 的标志性体验。用户会这样向朋友描述："扫个码就能进，点一下就下注了，所有人都能看到。"

### 用户心智模型

**用户目前的解决方案：**
- 纸笔记账：容易出错，不透明，结算时争议多
- 计算器：需要一个人专门负责，效率低
- 实体筹码：需要提前准备，不方便携带

**用户带来的心智模型：**
- 类似微信红包/AA收款的操作体验：扫码 → 操作 → 完成
- 期望"点一下就完成"，不需要复杂流程
- 期望所有人看到一致的结果

**用户可能困惑的地方：**
- "负数筹码是什么意思？" → 简单解释（你欠了多少）
- "平分池底怎么操作？" → 直观的选人界面
- "收回池底是谁收？" → 明确操作归属

### 核心交互成功标准

1. **速度**：下注操作从点击到所有人看到 ≤ 1 秒
2. **简单**：常用下注操作 ≤ 2 次点击（选金额 → 确认）
3. **信任**：所有人看到完全一致的筹码和池底状态
4. **可靠**：掉线后 5 秒内自动恢复，不丢失任何操作

### 新颖 vs. 成熟模式评估

| 交互 | 模式选择 | 理由 |
|------|----------|------|
| 加入房间 | 成熟（扫码进入） | 用户已习惯微信扫码，零学习成本 |
| 下注操作 | 成熟（快捷按钮） | 类似支付宝快捷金额，用户熟悉 |
| 实时同步 | 成熟（WebSocket推送） | 类似微信消息同步，用户期望如此 |
| 结算展示 | 成熟（列表+校验） | 类似AA收款结果，清晰直观 |
| 平分池底 | 微创新（选人+均分） | 简单的选人交互，逻辑直观 |

**结论：** 所有核心交互使用用户已熟悉的成熟模式，降低学习成本。

### 体验机制设计

**下注机制：**
1. 点击快捷按钮（20/50/100/1000）或输入自定义金额
2. 弹出确认："下注 100？"
3. 确认后：自己筹码 -100（动画）、池底 +100（动画）、所有人看到提示

**收回池底机制：**
1. 点击"收回池底"按钮
2. 确认后：池底归零、自己筹码 +池底总额（动画）、所有人看到提示

**平分池底机制：**
1. 点击"平分池底"按钮
2. 选择参与平分的玩家（勾选）
3. 系统自动计算每人分得金额
4. 确认后：池底归零、每人筹码 +均分金额（动画）、所有人看到提示

**结算机制：**
1. 房主点击"结束牌局"
2. 系统计算每人盈亏，执行总账校验（盈亏之和 = 0）
3. 显示结算页面：每人盈亏清单（正数绿色/负数红色）+ "总账校验通过 ✓"
4. 关闭后房间销毁，数据不保留

## 视觉设计基础

### 颜色系统

**主题：清新信任感**

基于 suoha 的情感目标（信任、放心、轻松），采用以下颜色方案：

| 角色 | 颜色 | 用途 |
|------|------|------|
| 主色 | `#2563EB`（蓝色） | 品牌色、主要按钮、导航栏 |
| 成功色 | `#16A34A`（绿色） | 赢钱、总账校验通过、正数筹码 |
| 警告色 | `#EF4444`（红色） | 输钱、负数筹码、错误状态 |
| 强调色 | `#F59E0B`（琥珀色） | 快捷下注按钮、重要操作 |
| 背景色 | `#F8FAFC`（浅灰白） | 页面背景 |
| 卡片色 | `#FFFFFF`（白色） | 卡片、面板背景 |
| 文字色 | `#1E293B`（深灰） | 主要文字 |
| 次要文字 | `#64748B`（中灰） | 辅助信息、提示文字 |
| 分割线 | `#E2E8F0`（浅灰） | 分割线、边框 |

**颜色策略：**
- 蓝色主色传达信任和可靠性（类似银行、支付应用）
- 绿色/红色清晰区分盈亏状态
- 琥珀色用于快捷操作，吸引注意但不过于刺激
- 高对比度确保移动端阳光下可读

**无障碍考虑：**
- 所有文字与背景对比度 ≥ 4.5:1（WCAG AA 标准）
- 不仅依赖颜色传达信息（盈亏同时用 +/- 符号）

### 字体系统

**策略：系统字体 + 等宽数字**

| 用途 | 字体 | 说明 |
|------|------|------|
| 正文 | 系统字体栈 | `-apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif` |
| 数字 | `"SF Mono", "Roboto Mono", monospace` | 筹码、池底等关键数字，等宽确保对齐 |

**字号层级：**

| 层级 | 大小 | 行高 | 用途 |
|------|------|------|------|
| 特大 | 32px | 1.2 | 池底总额 |
| 大 | 24px | 1.3 | 个人筹码数 |
| 中 | 16px | 1.5 | 快捷按钮文字、操作提示 |
| 小 | 14px | 1.5 | 辅助信息、玩家列表 |
| 极小 | 12px | 1.4 | 时间戳、状态提示 |

**字体策略：**
- 使用系统字体确保快速加载和原生体验
- 数字使用等宽字体，避免数字跳动（如 1111 vs 8888 宽度一致）
- 行高适中，确保移动端可读性

### 间距与布局基础

**间距系统（基于 4px 网格）：**

| Token | 值 | 用途 |
|-------|-----|------|
| xs | 4px | 紧凑间距 |
| sm | 8px | 元素内间距 |
| md | 16px | 卡片内间距、列表项间距 |
| lg | 24px | 区块间距 |
| xl | 32px | 页面边距 |

**触摸目标规范：**
- 最小触摸目标：44px × 44px（iOS 人机界面指南）
- 快捷下注按钮：56px 高度（大拇指友好）
- 按钮间距：≥ 8px（防止误触）

**布局原则：**

```
┌─────────────────────┐
│ 顶部栏：房间号 + 状态  │  44px（固定）
├─────────────────────┤
│                     │
│ 池底区域：池底总额     │  ~80px（固定）
│ + 每人下注明细        │
│                     │
├─────────────────────┤
│                     │
│ 玩家列表：            │  flex（可滚动）
│ 每人筹码 + 操作状态    │
│                     │
├─────────────────────┤
│ 操作区域：            │
│ 快捷按钮 + 自定义输入  │  ~120px（固定）
│ 收回/平分按钮         │
└─────────────────────┘
```

**关键布局策略：**
- 池底总额始终可见（顶部固定）
- 操作按钮始终可见（底部固定）
- 玩家列表可滚动（中间区域）
- 单手操作友好（操作区域在底部，拇指可达）
- 375px-428px 宽度优化（主流移动设备）

### 无障碍考虑

**颜色对比度：**
- 所有文字与背景对比度 ≥ 4.5:1（WCAG AA）
- 重要操作按钮对比度 ≥ 7:1（WCAG AAA）

**信息传达：**
- 不仅依赖颜色（盈亏同时用 +/- 符号和颜色）
- 操作反馈同时使用视觉和文字提示

**触摸友好：**
- 所有可点击元素 ≥ 44px × 44px
- 关键操作按钮更大（56px 高度）
- 按钮间距充足，防止误触

## 设计方向决策

### 设计方向探索

为了找到最适合"梭哈"应用的视觉设计方向，我们探索了 6 个不同的设计变体：

1. **极简专业型** - 大量留白，蓝色主色调，卡片式布局
2. **游戏化活力型** - 渐变色背景，圆角设计，emoji 元素
3. **深色专注型** - 深色背景，高对比度，适合夜间场景
4. **扁平简洁型** - 纯色块设计，无阴影，大按钮
5. **卡片层叠型** - 卡片式玩家列表，网格布局，视觉层次丰富
6. **列表紧凑型** - 紧凑布局，信息密度高，效率优先

每个方向都保持核心交互模式不变（扫码进入、快捷按钮下注、实时同步），但在视觉风格、信息密度、色彩运用上有所差异。

**完整设计方向展示：** `_bmad-output/planning-artifacts/ux-design-directions.html`

### 选定方向

**方向 1: 极简专业型**

这是最符合"梭哈"核心情感目标（信任、放心、专注游戏）的设计方向。

**核心视觉特征：**
- **留白充足** - 突出关键数字（池底、筹码），不干扰游戏进程
- **蓝色主色调** - #2563EB，传达信任和可靠性（类似银行、支付应用）
- **卡片式布局** - 清晰的信息分区，玩家列表用卡片分隔
- **简洁视觉语言** - 无过度装饰，专注于功能和可读性

**布局结构：**
```
┌─────────────────────┐
│ 顶部栏：房间号 + 状态  │  蓝色背景，白色文字
├─────────────────────┤
│ 池底区域：            │  浅灰背景 (#F8FAFC)
│ "池底" 标签           │  48px 大字号数字
│ 1,250               │  等宽字体，深色
├─────────────────────┤
│ 玩家列表：            │  白色卡片
│ [小李 (我)  +350]    │  边框分隔
│ [老王      -120]    │  绿色/红色数字
│ [小张        0]     │
├─────────────────────┤
│ 操作区域：            │  白色背景
│ [20][50][100][1000] │  琥珀色快捷按钮
│ [收回池底][平分池底]  │  蓝色操作按钮
└─────────────────────┘
```

### 设计理由

选择"极简专业型"的核心原因：

1. **建立信任感**
   - 蓝色主色调是金融和支付应用的标准色，用户已建立"蓝色 = 可靠"的心智模型
   - 专业的视觉风格传达"这是一个认真的记账工具，不是玩具"
   - 简洁的设计减少视觉噪音，让用户相信系统专注于核心功能

2. **易读性优先**
   - 留白充足，关键数字（池底、筹码）一眼可见
   - 高对比度确保移动端阳光下可读（线下游戏场景常见）
   - 等宽字体确保数字对齐，避免跳动

3. **不干扰游戏**
   - 简洁的界面不会分散玩家注意力
   - 用户可以专注于德州扑克游戏本身，而不是记账工具
   - 符合"即用即走"的产品定位

4. **通用性强**
   - 适合各种场合，从朋友聚会到正式牌局
   - 不会因为过于游戏化而显得不严肃
   - 不会因为过于紧凑而难以阅读

5. **符合用户心智模型**
   - 类似微信 AA 收款、支付宝群收款的视觉风格
   - 用户已熟悉这种"蓝色 + 卡片"的设计语言
   - 零学习成本

### 实现方案

基于选定的设计方向，具体实现要点：

**颜色应用：**
- 主色 #2563EB：顶部栏背景、操作按钮（收回/平分池底）
- 强调色 #F59E0B：快捷下注按钮（20/50/100/1000）
- 成功色 #16A34A：正数筹码、赢钱状态
- 警告色 #EF4444：负数筹码、输钱状态
- 背景色 #F8FAFC：池底区域背景
- 卡片色 #FFFFFF：玩家列表卡片、操作区域背景
- 边框色 #E2E8F0：卡片边框、分割线

**字体应用：**
- 池底数字：48px，等宽字体，深色 (#1E293B)
- 筹码数字：20px，等宽字体，绿色/红色/深色
- 按钮文字：16px，系统字体，白色
- 玩家名称：16px，系统字体，深色 (#1E293B)
- 辅助信息：14px，系统字体，中灰 (#64748B)

**间距应用：**
- 顶部栏：12px 内边距
- 池底区域：32px 上下内边距，16px 左右内边距
- 玩家列表：16px 外边距，每个卡片 16px 内边距，8px 间距
- 操作区域：16px 内边距
- 快捷按钮：8px 间距，16px 内边距
- 操作按钮：8px 间距，14px 内边距

**交互反馈：**
- 按钮点击：按下效果（轻微缩放或背景变暗）
- 数字变化：动画过渡（0.3s ease-out）
- 操作提示：顶部 toast 消息（3s 自动消失）
- 连接状态：顶部栏右侧显示"● 已连接"（绿色）或"● 重连中"（黄色）

**响应式策略：**
- 375px-428px 宽度优化（主流移动设备）
- 池底和操作区域固定高度，玩家列表可滚动
- 单手操作友好，操作按钮在底部（拇指可达）
- 横屏模式：保持竖屏布局，不做特殊适配（德州扑克通常竖屏玩）

**深色模式考虑：**
- 未来可考虑添加深色模式（参考方向 3: 深色专注型）
- 适合夜间游戏场景，减少视觉疲劳
- 保持相同的布局和交互，仅调整配色
